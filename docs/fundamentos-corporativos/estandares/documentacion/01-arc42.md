# Estándar: arc42 - Plantilla de Documentación

## 1. Propósito

Establecer **arc42** como la plantilla estándar para documentar arquitecturas de software en Talma, proporcionando una estructura consistente, completa y comprensible para todos los stakeholders técnicos y de negocio.

:::tip Complementariedad
Usa arc42 para la **estructura de documentación** y [C4 Model](./02-c4-model.md) para los **diagramas arquitectónicos**. Ambos estándares se complementan perfectamente.
:::

## 2. Alcance

### 2.1 Aplicaciones Objetivo

- ✅ Documentación de arquitectura de sistemas nuevos
- ✅ Actualización de documentación existente
- ✅ Comunicación entre equipos técnicos y no técnicos
- ✅ Onboarding de nuevos desarrolladores
- ✅ Auditorías y compliance (SOC 2, ISO 27001)
- ✅ Revisiones de diseño y arquitectura

### 2.2 Uso Obligatorio

- Sistemas críticos de negocio
- Aplicaciones multi-equipo (> 5 desarrolladores)
- Proyectos con duración estimada > 6 meses
- Sistemas con integraciones complejas (> 3 sistemas externos)

### 2.3 Uso Recomendado

- Microservicios y sistemas distribuidos
- Migraciones y modernizaciones de legacy
- POCs que puedan evolucionar a producción

### 4.2 Sección 1: Introducción y Objetivos

**Contenido**:
- Propósito del sistema
- Stakeholders principales
- Objetivos de negocio y técnicos
- Features principales

**Ejemplo**:

```markdown
## 1. Introducción y Objetivos

### Propósito

Sistema de Gestión de Usuarios (SGU) permite la administración centralizada de
usuarios, roles y permisos para todas las aplicaciones de Talma.

### Stakeholders

| Rol | Nombre | Expectativas |
|-----|--------|--------------|
| Usuarios finales | Empleados de Talma | Acceso rápido y simple a aplicaciones |
| Administradores | Equipo de IT | Gestión centralizada de identidades |
| Desarrolladores | Equipos de desarrollo | API clara para integración SSO |
| Compliance | Auditoría interna | Trazabilidad de accesos |

### Objetivos de Negocio

- Centralizar gestión de identidades (reducir de 5 sistemas a 1)
- Reducir tiempo de onboarding de usuarios (de 2 días a 2 horas)
- Cumplir con SOC 2 y ISO 27001
- Reducir costos de licenciamiento (consolidar proveedores)

### Objetivos Técnicos

- Single Sign-On (SSO) con OAuth 2.0 / OpenID Connect
- Alta disponibilidad (99.9% uptime SLA)
- Tiempo de respuesta < 200ms (p95)
- Escalabilidad horizontal (10K usuarios concurrentes)
```

```markdown
- **Responsabilidad**: Endpoints HTTP para CRUD de usuarios

- **Responsabilidad**: Lógica de negocio de usuarios

- **Responsabilidad**: Acceso a datos de usuarios
- **Tecnología**: Entity Framework Core 8.0
```

### 4.6 Sección 8: Conceptos Transversales

**Contenido**:
- Seguridad (autenticación, autorización, cifrado)
- Persistencia (estrategia de datos)
- Logging y observabilidad
- Manejo de errores
- UI/UX patterns

**Ejemplo**:

```markdown
## 8. Conceptos Transversales

### Seguridad

**Autenticación**:
- OAuth 2.0 + OpenID Connect
- JWT Bearer tokens (RS256)
- Access token: 15 minutos TTL
- Refresh token: 7 días TTL (rotation)

**Autorización**:
- RBAC (Role-Based Access Control)
- Claims-based (email, name, role, country)
- Policy-based con `[Authorize(Policy = "AdminOnly")]`

**Cifrado**:
- TLS 1.3 para datos en tránsito
- AES-256 para datos en reposo (RDS encryption)
- Passwords con bcrypt (work factor 12)

**Referencias**:
- [Estándar: Seguridad de APIs](../apis/02-seguridad-apis.md)
- [ADR-004: Autenticación SSO](../../decisiones-de-arquitectura/adr-004-autenticacion-sso.md)

### Persistencia

**Base de datos primaria**: PostgreSQL 16 (RDS Multi-AZ)

**Estrategia**:
- Entity Framework Core con Code-First migrations
- Repository pattern para data access
- Unit of Work para transacciones
- Read replicas para consultas de solo lectura

**Referencias**:
- [Estándar: SQL](../codigo/03-sql.md)
- [ADR-010: Estándar de Base de Datos](../../decisiones-de-arquitectura/adr-010-standard-base-datos.md)

### Logging

**Framework**: Serilog 3.1 con structured logging

**Niveles**:
- `Error`: Errores no recuperables
- `Warning`: Errores recuperables, degradación
- `Information`: Eventos de negocio (UserCreated, LoginSuccessful)
- `Debug`: Información técnica (solo dev/staging)

**Enrichers**:
- `CorrelationId`: Trazabilidad de requests
- `UserId`: Usuario que ejecuta la acción
- `Environment`: dev/staging/prod
- `Version`: Versión del servicio

**Referencias**:
- [Estándar: Logging](../observabilidad/01-logging.md)
- [ADR-016: Logging Estructurado](../../decisiones-de-arquitectura/adr-016-logging-estructurado.md)
```

```markdown
## 9. Decisiones de Arquitectura

### ADRs Aplicables

| ADR | Título | Decisión | Impacto |
|-----|--------|----------|---------|
| [ADR-001](../../decisiones-de-arquitectura/adr-001-multi-tenancy-paises.md) | Multi-Tenancy por País | Datos separados por país | Alto |
| [ADR-002](../../decisiones-de-arquitectura/adr-002-estandard-apis-rest.md) | Estándar de APIs REST | REST sobre GraphQL | Medio |
| [ADR-004](../../decisiones-de-arquitectura/adr-004-autenticacion-sso.md) | Autenticación SSO | OAuth 2.0 + OpenID Connect | Alto |
| [ADR-010](../../decisiones-de-arquitectura/adr-010-standard-base-datos.md) | Base de Datos | PostgreSQL sobre DynamoDB | Alto |

### Decisiones Clave

#### JWT vs Session Cookies

**Contexto**: Necesidad de autenticación stateless para APIs

**Decisión**: JWT Bearer tokens (access + refresh token)

**Alternativas consideradas**:
- Session cookies: Descartado (stateful, difícil de escalar)
- API keys: Descartado (sin expiración automática)

**Consecuencias**:
- ✅ Stateless (facilita escalado horizontal)
- ✅ Compatible con SPAs y apps móviles
- ⚠️ Complejidad de rotación de refresh tokens
- ⚠️ No se pueden revocar tokens antes de expiración

**Referencias**: [ADR-004](../../decisiones-de-arquitectura/adr-004-autenticacion-sso.md)
```

## 6. Antipatrones

### 6.1 ❌ Documentación Monolítica

**Problema**:
```markdown
# ❌ Un solo archivo de 5000 líneas
arquitectura.md (5000 líneas)
- Imposible de navegar
- Conflictos en Git
- Desmotiva actualización
```

**Solución**:
```markdown
# ✅ Separado en 12 archivos
docs/arquitectura/
├── 01-introduccion-objetivos.md    (200 líneas)
├── 02-restricciones.md             (150 líneas)
├── 03-contexto-alcance.md          (300 líneas)
...
└── 12-glosario.md                   (100 líneas)
```

### 6.2 ❌ Documentar el Futuro

**Problema**:
```markdown
## 5. Vista de Bloques

### Sistema de Notificaciones (Planeado para Q3 2026)
[Diagrama de sistema que no existe aún]
```

**Solución**:
```markdown
## 5. Vista de Bloques

### Sistemas Actuales
[Solo lo que está en producción o staging]

## 11. Riesgos y Deuda Técnica

### Roadmap Futuro
- Q3 2026: Sistema de notificaciones (ver ADR-025 draft)
```

### 6.3 ❌ Documentación Desactualizada

**Problema**:
```markdown
<!-- ❌ Última actualización: 2023-01-15 -->
<!-- Código actual usa .NET 8, documentación dice .NET 6 -->
```

**Solución**:
```markdown
# ✅ Agregar en CI/CD
- name: Check docs freshness
  run: |
    LAST_UPDATE=$(git log -1 --format=%cd --date=short docs/)
    if [ $(date -d "$LAST_UPDATE" +%s) -lt $(date -d "3 months ago" +%s) ]; then
      echo "⚠️ Documentación desactualizada (> 3 meses)"
      exit 1
    fi
```

### 6.4 ❌ Omitir Secciones Sin Justificación

**Problema**:
```markdown
# ❌ Saltar secciones porque "no aplican"
## 1. Introducción ✅
## 2. Restricciones ❌ (omitida)
## 3. Contexto ✅
...
```

**Solución**:
```markdown
# ✅ Explicar explícitamente por qué se omite
## 2. Restricciones

Esta sección no aplica para este POC de 2 semanas porque:
- No hay restricciones técnicas (stack libre)
- No hay restricciones legales (datos sintéticos)
- Timeline flexible (exploración)

**Nota**: Si el POC evoluciona a producción, esta sección debe completarse.
```

### 7.2 Validación Automática en CI/CD

```yaml
# .github/workflows/docs-validation.yml
name: Validate arc42 Documentation

on:
  pull_request:
    paths:
      - 'docs/arquitectura/**'

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Check all sections exist
        run: |
          SECTIONS=(
            "01-introduccion-objetivos.md"
            "02-restricciones.md"
            "03-contexto-alcance.md"
            "04-estrategia-solucion.md"
            "05-vista-bloques.md"
            "06-vista-runtime.md"
            "07-vista-despliegue.md"
            "08-conceptos-transversales.md"
            "09-decisiones-arquitectura.md"
            "10-requisitos-calidad.md"
            "11-riesgos-deuda-tecnica.md"
            "12-glosario.md"
          )
          
          for section in "${SECTIONS[@]}"; do
            if [ ! -f "docs/arquitectura/$section" ]; then
              echo "❌ Falta sección: $section"
              exit 1
            fi
          done
          echo "✅ Todas las secciones presentes"
      
      - name: Check links validity
        uses: gaurav-nelson/github-action-markdown-link-check@v1
        with:
          use-quiet-mode: 'yes'
          config-file: '.markdown-link-check.json'
      
      - name: Check freshness
        run: |
          LAST_UPDATE=$(git log -1 --format=%cd --date=short docs/arquitectura/)
          DAYS_OLD=$(( ($(date +%s) - $(date -d "$LAST_UPDATE" +%s)) / 86400 ))
          
          if [ $DAYS_OLD -gt 90 ]; then
            echo "⚠️ Documentación desactualizada ($DAYS_OLD días)"
            exit 1
          fi
          echo "✅ Documentación actualizada ($DAYS_OLD días)"
```
