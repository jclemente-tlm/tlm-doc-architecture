# Estándar: arc42 - Plantilla de Documentación

## 1. Propósito

Establecer **arc42** como la plantilla estándar para documentar arquitecturas de software en Talma, proporcionando una estructura consistente, completa y comprensible para todos los stakeholders.

## 2. Alcance

- Documentación de arquitectura de sistemas nuevos
- Actualización de documentación existente
- Comunicación entre equipos técnicos y no técnicos
- Onboarding de nuevos desarrolladores
- Auditorías y compliance

## 3. ¿Qué es arc42?

**arc42** es una plantilla para documentación de arquitecturas de software con **12 secciones** que cubren todos los aspectos relevantes de un sistema, desde su contexto hasta decisiones técnicas y riesgos.

**Ventajas**:

- ✅ Estructura probada y estandarizada
- ✅ Flexible (usar solo secciones necesarias)
- ✅ Independiente de tecnología
- ✅ Open source y gratuita
- ✅ Soporta múltiples formatos (Markdown, AsciiDoc, Word)

## 4. Las 12 Secciones de arc42

### 📋 Plantilla Completa

| #   | Sección                    | Propósito                                          | Audiencia           |
| --- | -------------------------- | -------------------------------------------------- | ------------------- |
| 1   | Introducción y Objetivos   | ¿Qué problema resuelve? ¿Cuáles son los objetivos? | Todos               |
| 2   | Restricciones              | Limitaciones técnicas, organizacionales, legales   | Arquitectos, PM     |
| 3   | Contexto y Alcance         | Fronteras del sistema, interfaces externas         | Todos               |
| 4   | Estrategia de Solución     | Decisiones arquitectónicas clave                   | Arquitectos         |
| 5   | Vista de Bloques           | Estructura interna (contenedores, componentes)     | Arquitectos, Devs   |
| 6   | Vista de Runtime           | Comportamiento en tiempo de ejecución              | Arquitectos, Devs   |
| 7   | Vista de Despliegue        | Infraestructura, ambientes, hardware               | DevOps, Arquitectos |
| 8   | Conceptos Transversales    | Seguridad, persistencia, UI, logging               | Todos               |
| 9   | Decisiones de Arquitectura | ADRs (Architecture Decision Records)               | Arquitectos         |
| 10  | Requisitos de Calidad      | Performance, seguridad, disponibilidad             | Todos               |
| 11  | Riesgos y Deuda Técnica    | Problemas conocidos, limitaciones                  | Arquitectos, PM     |
| 12  | Glosario                   | Términos del dominio                               | Todos               |

## 5. Detalle de Secciones Clave

### 5.1 Introducción y Objetivos

**Contenido**:

- Propósito del sistema
- Stakeholders principales
- Objetivos de negocio y técnicos
- Features principales

**Ejemplo**:

```markdown
## 1. Introducción y Objetivos

### Propósito

Sistema de Gestión de Usuarios (SGU) permite la administración centralizada de
usuarios, roles y permisos para todas las aplicaciones de Talma.

### Stakeholders

- **Usuarios finales**: Empleados de Talma
- **Administradores**: Equipo de IT
- **Desarrolladores**: Equipos de desarrollo de aplicaciones
- **Compliance**: Auditoría interna

### Objetivos de Negocio

- Centralizar gestión de identidades
- Reducir tiempo de onboarding de usuarios
- Cumplir con SOC 2 y ISO 27001

### Objetivos Técnicos

- Single Sign-On (SSO) con OAuth 2.0
- Alta disponibilidad (99.9% uptime)
- Tiempo de respuesta < 200ms (p95)
```

### 5.2 Restricciones

**Contenido**:

- Restricciones técnicas (tecnologías obligatorias, legacy)
- Restricciones organizacionales (presupuesto, tiempo, equipo)
- Restricciones legales/compliance

**Ejemplo**:

```markdown
## 2. Restricciones

### Técnicas

- **Cloud Provider**: AWS (mandatorio por política corporativa)
- **Base de datos**: PostgreSQL 14+ (estándar corporativo)
- **Lenguaje backend**: C# / .NET 8 (stack principal)
- **Integración legacy**: SOAP API existente (no se puede eliminar)

### Organizacionales

- **Timeline**: MVP en 3 meses
- **Team size**: 4 desarrolladores, 1 QA, 1 DevOps
- **Budget**: $50K para infraestructura anual

### Legales

- **GDPR**: Datos personales deben poder eliminarse
- **SOC 2**: Logs de auditoría obligatorios
- **Residencia de datos**: Datos de Perú deben estar en us-east-1
```

### 5.3 Contexto y Alcance

**Contenido**:

- Diagrama de contexto ([C4 Level 1](./02-c4-model.md))
- Interfaces externas (APIs, sistemas legacy)
- Usuarios/actores del sistema

**Ejemplo**:

```markdown
## 3. Contexto y Alcance

### Diagrama de Contexto

[Incluir diagrama C4 Context - ver estándar C4 Model]

### Interfaces Externas

| Sistema Externo          | Tipo     | Propósito                   |
| ------------------------ | -------- | --------------------------- |
| Active Directory         | LDAP     | Autenticación corporativa   |
| SAP HR                   | REST API | Sincronización de empleados |
| Salesforce               | Webhook  | Notificación de cambios     |
| Email Service (SendGrid) | REST API | Envío de notificaciones     |

### Actores

- **Usuario final**: Empleado que inicia sesión en aplicaciones
- **Administrador IT**: Gestiona usuarios y permisos
- **Aplicación cliente**: Sistema que consume SSO
```

### 5.4 Estrategia de Solución

**Contenido**:

- Patrones arquitectónicos elegidos
- Descomposición del sistema
- Decisiones tecnológicas clave

**Ejemplo**:

```markdown
## 4. Estrategia de Solución

### Arquitectura de Alto Nivel

- **Patrón**: Clean Architecture con DDD
- **Estilo**: API-first, microservicios (preparado para escalar)
- **Comunicación**: REST APIs con OpenAPI

### Decisiones Clave

- **Autenticación**: JWT Bearer tokens con refresh token rotation
- **Autorización**: RBAC (Role-Based Access Control) con claims
- **Cache**: Redis distribuido para sesiones
- **Base de datos**: PostgreSQL con replicación read replicas

### Referencias a ADRs

- [ADR-001: Usar JWT en lugar de session cookies](../../decisiones-de-arquitectura/adr-001-jwt-tokens.md)
- [ADR-002: PostgreSQL sobre DynamoDB](../../decisiones-de-arquitectura/adr-002-postgresql.md)
```

### 5.5 Vista de Bloques

**Contenido**:

- **Nivel 1**: Contenedores ([C4 Level 2](./02-c4-model.md))
- **Nivel 2**: Componentes dentro de contenedores ([C4 Level 3](./02-c4-model.md))
- Responsabilidades de cada bloque

**Ejemplo**:

```markdown
## 5. Vista de Bloques

### Nivel 1: Contenedores

[Incluir diagrama C4 Container]

#### Web API (ASP.NET Core)

**Responsabilidad**: Exponer endpoints REST para operaciones de usuarios

**Interfaces**:

- REST API: `https://api.talma.com/users` (HTTPS:443)
- Health check: `/health`, `/health/ready`, `/health/live`

**Tecnologías**:

- ASP.NET Core 8
- Entity Framework Core 8
- Serilog para logging
- FluentValidation

**Dependencias**:

- PostgreSQL (data persistence)
- Redis (session cache)
- SendGrid (email notifications)

**Código fuente**: [GitHub](https://github.com/talma/users-api)

### Nivel 2: Componentes de Web API

[Incluir diagrama C4 Component]

#### User Controller

- **Responsabilidad**: Endpoints HTTP para CRUD de usuarios
- **Tecnología**: ASP.NET Core MVC Controllers
- **Expone**: GET/POST/PUT/DELETE `/api/v1/users`

#### User Service

- **Responsabilidad**: Lógica de negocio de usuarios
- **Tecnología**: C# class library
- **Reglas**: Validación de email único, password policy

#### User Repository

- **Responsabilidad**: Acceso a datos de usuarios
- **Tecnología**: Entity Framework Core
- **Patrón**: Repository pattern con Unit of Work
```

### 5.6 Vista de Runtime

**Contenido**:

- Diagramas de secuencia
- Flujos de casos de uso importantes
- Comportamiento en escenarios específicos

**Ejemplo**:

```markdown
## 6. Vista de Runtime

### Caso de Uso: Login de Usuario

**Secuencia**:

1. Usuario envía credenciales a `/api/v1/auth/login`
2. API valida formato (FluentValidation)
3. API consulta PostgreSQL para verificar usuario
4. API valida password con bcrypt
5. API genera JWT access token (15min) y refresh token (7 días)
6. API guarda refresh token en Redis
7. API retorna tokens al cliente
8. Cliente almacena tokens en localStorage (SPA) o httpOnly cookie (SSR)

**Diagrama de secuencia**:

\`\`\`mermaid
sequenceDiagram
participant U as Usuario
participant API as Web API
participant DB as PostgreSQL
participant Cache as Redis

    U->>API: POST /auth/login {email, password}
    API->>API: Validar formato
    API->>DB: SELECT user WHERE email = ?
    DB-->>API: User data
    API->>API: Validar password (bcrypt)
    API->>API: Generar JWT + refresh token
    API->>Cache: SETEX refresh_token
    Cache-->>API: OK
    API-->>U: {accessToken, refreshToken}

\`\`\`

### Caso de Uso: Token Refresh

[Similar estructura]
```

### 5.8 Conceptos Transversales

**Contenido**:

- Seguridad
- Persistencia
- Manejo de errores
- Logging y monitoreo
- UI/UX patterns

**Ejemplo**:

```markdown
## 8. Conceptos Transversales

### Seguridad

- **Autenticación**: JWT Bearer tokens
- **Autorización**: Claims-based con políticas
- **Encriptación**: TLS 1.3 en tránsito, AES-256 en reposo
- **Secrets**: AWS Secrets Manager con rotación automática
- **OWASP Top 10**: Mitigaciones implementadas
  - SQL Injection: Parametrized queries (EF Core)
  - XSS: Content-Security-Policy headers
  - CSRF: SameSite cookies

Ver: [Estándar de Seguridad de APIs](../apis/02-seguridad-apis.md)

### Persistencia

- **Patrón**: Repository + Unit of Work
- **ORM**: Entity Framework Core
- **Migraciones**: Code-first migrations versionadas
- **Backup**: Snapshots diarios, retention 30 días

### Logging

- **Framework**: Serilog con structured logging
- **Destino**: CloudWatch Logs
- **Nivel**: Information (production), Debug (dev)
- **Correlation ID**: X-Correlation-Id header propagado

Ver: [Estándar de Logging](../observabilidad/01-logging.md)

### Manejo de Errores

- **Formato**: RFC 7807 (Problem Details)
- **Global exception handler**: Middleware custom
- **Códigos HTTP**: Usar correctamente (200, 400, 401, 404, 500)

Ver: [Estándar de Validación y Errores](../apis/03-validacion-y-errores.md)
```

### 5.10 Requisitos de Calidad

**Contenido**:

- Atributos de calidad (ISO 25010)
- SLOs/SLAs
- NFRs (Non-Functional Requirements)

**Ejemplo**:

```markdown
## 10. Requisitos de Calidad

### Performance

- **Latencia**: p95 < 200ms, p99 < 500ms
- **Throughput**: 1000 req/s
- **Tiempo de respuesta DB**: < 50ms

### Disponibilidad

- **SLA**: 99.9% uptime (43.8 min downtime/mes)
- **RPO**: 1 hora (Recovery Point Objective)
- **RTO**: 30 minutos (Recovery Time Objective)

### Seguridad

- **Autenticación**: Multi-factor (MFA) obligatorio para admins
- **Sesiones**: Expiración automática después de 15 minutos inactividad
- **Auditoría**: Logs de todas las operaciones sensibles

### Escalabilidad

- **Horizontal**: Auto-scaling hasta 10 tasks
- **Vertical**: Soportar 10K usuarios concurrentes
- **Database**: Read replicas para queries read-heavy

### Mantenibilidad

- **Cobertura de tests**: > 80%
- **Deuda técnica**: < 1 día por sprint
- **Code smells**: Sonarqube Quality Gate "passed"
```

### 5.11 Riesgos y Deuda Técnica

**Contenido**:

- Riesgos conocidos
- Deuda técnica identificada
- Plan de mitigación

**Ejemplo**:

```markdown
## 11. Riesgos y Deuda Técnica

### Riesgos

| Riesgo                                      | Probabilidad | Impacto | Mitigación                                 |
| ------------------------------------------- | ------------ | ------- | ------------------------------------------ |
| Integración con SAP HR falla frecuentemente | Alta         | Medio   | Implementar circuit breaker y retry policy |
| PostgreSQL single point of failure          | Baja         | Alto    | Multi-AZ con failover automático           |
| Límite de rate de SendGrid                  | Media        | Bajo    | Implementar queue con SQS para emails      |

### Deuda Técnica

| Item                                            | Esfuerzo  | Prioridad | Plan          |
| ----------------------------------------------- | --------- | --------- | ------------- |
| Migrar de REST a gRPC para comunicación interna | 2 semanas | Media     | Q2 2026       |
| Implementar CQRS para reportes                  | 1 semana  | Baja      | Q3 2026       |
| Refactor de UserService (300 líneas)            | 3 días    | Alta      | Sprint actual |
```

## 7. Checklist de Documentación

Al documentar un sistema con arc42, verificar:

- [ ] **Sección 1**: Objetivos claros y stakeholders identificados
- [ ] **Sección 2**: Restricciones documentadas (técnicas, organizacionales, legales)
- [ ] **Sección 3**: Diagrama de contexto actualizado (C4 Level 1)
- [ ] **Sección 4**: Estrategia de solución con referencias a ADRs
- [ ] **Sección 5**: Diagramas de contenedores y componentes (C4 Level 2-3)
- [ ] **Sección 6**: Al menos 2 flujos de runtime documentados
- [ ] **Sección 7**: Deployment actualizado con ambientes
- [ ] **Sección 8**: Conceptos transversales con referencias a estándares
- [ ] **Sección 9**: ADRs versionados y linkados
- [ ] **Sección 10**: SLOs/SLAs definidos
- [ ] **Sección 11**: Riesgos y deuda técnica actualizados
- [ ] **Sección 12**: Glosario con términos del dominio
- [ ] **Versionado**: Documentación en Git con historial
- [ ] **Actualización**: Fecha de última revisión < 3 meses

## 8. NO Hacer

❌ **NO** crear toda la documentación de una vez (hacerlo incremental)
❌ **NO** documentar implementación detallada (código ya lo hace)
❌ **NO** usar diagramas como imágenes binarias (preferir texto/DSL)
❌ **NO** documentar "el futuro" (solo estado actual)
❌ **NO** omitir secciones relevantes sin justificación
❌ **NO** dejar documentación desactualizada > 6 meses
