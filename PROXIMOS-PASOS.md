## ✏️ Cómo Completar un Template

### 1. **Contexto**

Describir:

- ¿Qué problema resuelve este estándar?
- ¿En qué escenarios se aplica?
- ¿Por qué es necesario?

**Ejemplo (circuit-breakers):**

> Los servicios distribuidos pueden experimentar fallos en cascada cuando un servicio dependiente falla. Sin protección, todas las solicitudes se quedan bloqueadas esperando respuesta de un servicio no disponible, causando agotamiento de recursos y fallo total del sistema.

### 2. **Decisión**

- Mantener la frase base
- Opcional: Agregar detalles de la decisión específica

### 3. **Estándares Obligatorios**

Definir reglas concretas y medibles:

- Usar subsecciones numeradas (### 1., ### 2., etc.)
- Incluir configuraciones específicas
- Agregar ejemplos de código cuando aplique
- Definir umbrales, timeouts, límites

**Ejemplo (circuit-breakers):**

```markdown
### 1. Configuración de Circuit Breaker

Todos los llamados a servicios externos DEBEN protegerse con circuit breakers:

- **Failure Threshold:** 50% (de 10 requests mínimo)
- **Timeout:** 5 segundos
- **Sleep Window:** 60 segundos

### 2. Estados y Transiciones

- **Closed:** Operación normal, monitorear fallos
- **Open:** Fallos exceden threshold, rechazar requests inmediatamente
- **Half-Open:** Después de sleep window, permitir 1 request de prueba
```

### 4. **Validación de Cumplimiento**

Definir checklist verificable:

- [ ] Criterios específicos y medibles
- [ ] Incluir herramientas de validación
- [ ] Agregar métricas esperadas

**Ejemplo:**

```markdown
- [ ] Todos los clientes HTTP configuran circuit breaker con thresholds definidos
- [ ] Métricas de circuit breaker expuestas en Prometheus
- [ ] Alertas configuradas para estado "Open" por >5 minutos
```

### 5. **Referencias**

Enlazar a:

- Documentación oficial de frameworks mencionados
- RFCs, especificaciones técnicas
- Artículos técnicos relevantes
- Ejemplos de implementación
